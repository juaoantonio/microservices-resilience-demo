diagrama · circuit breaker

Circuit Breaker

componentes

Serviço Chamador Recebe a requisição e inicia a chamada externa
→
Circuit Breaker Mede falhas, latência e mantém o estado
→
Cliente Resiliente Camada que encapsula a chamada para a dependência
→
Dependência Externa API, banco ou serviço remoto sujeito a erro ou lentidão

fluxo de estado

closed

Encaminha chamadas e acumula métricas de sucesso, erro e slow call.

open

Ao atingir o limiar de falha, corta a chamada e responde via fallback.

half-open

Libera poucas chamadas de teste para decidir se fecha ou reabre.

falhas acima do limiar → abre circuito → fallback imediato
responsabilidades
  • Evitar cascata quando a dependência já está degradada
  • Falhar rápido para preservar threads e latência
  • Centralizar a decisão de reabrir ou retestar o serviço
  • Expor um ponto claro para fallback e observabilidade

O disjuntor não corrige a dependência. Ele protege o restante do sistema.

diagrama · retry

Retry

componentes

Serviço Chamador Dispara uma operação que pode falhar temporariamente
→
Retry Policy Controla número de tentativas e intervalo entre elas
→
Cliente Resiliente Repete a chamada somente nos erros configurados
→
Dependência Externa Falha intermitente que pode se recuperar na próxima chamada

fluxo de execução

tentativa 1

Se a resposta falhar com erro transitório, a política agenda nova chamada.

backoff

Espera controlada evita tempestade de novas requisições.

limite

Quando esgota as tentativas, encerra o fluxo e deixa o fallback assumir.

falha transitória → aguarda 500ms → reenvia → sucesso ou fallback
responsabilidades
  • Recuperar falhas temporárias sem intervenção manual
  • Limitar tentativas para não amplificar a indisponibilidade
  • Aplicar intervalo ou backoff entre chamadas
  • Exigir idempotência para não duplicar efeitos colaterais

Retry funciona bem para ruído temporário. Em erro persistente, ele deve parar cedo.

diagrama · timeout

Timeout

componentes

Requisição Entra no serviço com um orçamento máximo de tempo
→
TimeLimiter Relógio de proteção que observa a duração da chamada
→
Execução Assíncrona Executa a chamada externa sem bloquear indefinidamente
→
Fallback Assume quando o prazo estoura e devolve resposta controlada

fluxo de prazo

início

A chamada começa com budget fixo, por exemplo 2 segundos.

até o prazo

Se a resposta volta dentro do limite, o fluxo segue normalmente.

estouro

Ao exceder o tempo, a espera é cancelada e o fallback devolve resposta controlada.

inicia relógio → espera até 2s → cancela → retorna degradado
responsabilidades
  • Definir limite máximo aceitável para cada dependência
  • Impedir espera indefinida e prender menos recursos
  • Transformar lentidão em resposta previsível
  • Dar insumo para métricas de latência e SLA

Timeout converte lentidão em decisão explícita. Sem ele, a fila inteira pode parar junto.

diagrama · bulkhead

Bulkhead

componentes

Tráfego de Entrada Pico de requisições concorrentes chega ao serviço
→
Bulkhead Reserva poucas threads ou permissões só para essa integração
→
Cliente Resiliente Executa somente o volume permitido de chamadas
→
Dependência Externa Parte lenta ou congestionada que não deve contaminar o resto

fluxo de isolamento

capacidade

Somente um número fixo de chamadas entra no compartimento protegido.

excesso

O restante espera pouco ou é rejeitado imediatamente, sem drenar o pool global.

contenção

A falha fica isolada nessa integração e outros endpoints continuam saudáveis.

4 permissões ocupadas → 5ª chamada bloqueada → rejeição rápida
responsabilidades
  • Separar recursos críticos por dependência ou operação
  • Conter starvation de threads e saturação do pool
  • Definir concorrência máxima segura
  • Preservar estabilidade do serviço mesmo sob pico ou lentidão

Bulkhead aceita perder uma parte do tráfego para impedir que todo o navio afunde.

solução

Testcontainers 

@Container
static PostgreSQLContainer<?> postgres
    = new PostgreSQLContainer("postgres:16-alpine");

static WireMockServer wireMock; // simula payment-service

@DynamicPropertySource
static void configureProperties(DynamicPropertyRegistry r) {
    r.add("spring.datasource.url", postgres::getJdbcUrl);
    r.add("payment.base-url", () -> "http://localhost:" + wireMock.port());
}

Agora conseguimos trazer o caos da produção… para dentro do teste.

resiliência · banco de dados

ResilientDatabaseService 

@CircuitBreaker(name = "database", fallbackMethod = "saveFallback")
@Retry(name = "database")
public Order save(Order order) {
    return orderRepository.save(order);
}

public Order saveFallback(Order order, Throwable ex) {
    throw new DatabaseUnavailableException("Database unavailable", ex);
}
sem @TimeLimiter
Não usa CompletableFuture — incompatível com @Transactional síncrono
retry seletivo
Só TransientDataAccessException — deadlocks, pool esgotado, timeout de query

Mesmo padrão, nova camada — banco também precisa de proteção.
